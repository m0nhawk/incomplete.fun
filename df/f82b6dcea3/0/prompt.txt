# Push Committed Changes

You are helping push existing commits from the current git branch.

## Guardrails

- Do not create commits, stage files, edit files, or discard changes.
- Do not force-push unless the user explicitly asks for a force push.
- If the working tree has uncommitted changes, mention them before pushing. If the user's wording suggests those changes should be included, ask whether to commit first instead of pushing.
- Stop and ask if the repository is in a detached HEAD state or if the target remote/branch is unclear.

## Workflow

1. Inspect repository and branch state:
   - `git status --short --branch`
   - `git branch --show-current`
   - `git remote -v`
2. Identify the current branch and upstream:
   - Use `git rev-parse --abbrev-ref --symbolic-full-name @{u}` to check for an upstream.
   - If no upstream exists, plan to push to `origin/<current-branch>` unless the user specified another remote or branch.
3. Review what will be pushed:
   - With an upstream: `git log --oneline @{u}..HEAD`
   - Without an upstream: `git log --oneline origin/<current-branch>..HEAD` if that remote branch exists; otherwise show recent commits with `git log --oneline -5`.
4. If there are no commits ahead and an upstream exists, report that there is nothing to push and stop.
5. Push:
   - If upstream exists: `git push`
   - If upstream is missing: `git push -u origin <current-branch>`
6. Report:
   - remote and branch pushed
   - whether upstream was set
   - any uncommitted local changes that remain
</skill>

# Commit Local Changes

You are helping create a git commit for the current repository.

## Guardrails

- Do not push.
- Do not discard, revert, reset, or overwrite user changes.
- Do not commit secrets, credentials, editor junk, build artifacts, dependency folders, or ignored files unless the user explicitly requests it.
- Preserve intentional staging: if files are already staged, commit the staged changes and only add more files when the user's request clearly includes them.
- If staging is ambiguous, ask the user instead of guessing.

## Workflow

1. Inspect repository state:
   - `git status --short --branch`
   - `git diff --stat`
   - `git diff`
   - `git diff --staged --stat`
   - `git diff --staged`
2. If there are no staged or unstaged changes, report that there is nothing to commit and stop.
3. Decide what to stage:
   - If changes are already staged, keep the staged set unless the user asked to include additional files.
   - If nothing is staged, stage the appropriate tracked and untracked files with `git add -- <paths>`.
   - Prefer path-specific `git add -- <paths>` over broad `git add .`.
4. Re-check exactly what will be committed:
   - `git status --short`
   - `git diff --staged --stat`
   - `git diff --staged`
5. Choose a concise commit message based on the staged diff:
   - Follow the repository's existing style if obvious from recent commits (`git log --oneline -5`).
   - Otherwise use a short imperative summary, optionally with a body for multi-area changes.
6. Run `git commit -m "<message>"` (use additional `-m` lines for a body when useful).
7. Report:
   - commit hash (`git rev-parse --short HEAD`)
   - commit message
   - any uncommitted changes left behind
</skill>

# Ship Changes

You are helping commit local changes and push them to the remote branch.

## Guardrails

- Do not discard, revert, reset, or overwrite user changes.
- Do not commit secrets, credentials, editor junk, build artifacts, dependency folders, or ignored files unless explicitly requested.
- Preserve intentional staging: if files are already staged, commit the staged changes and only add more when the request clearly includes them.
- Do not force-push unless explicitly requested.
- Ask the user when staging, branch, remote, or push target is ambiguous.

## Workflow

### 1. Inspect changes

Run:

- `git status --short --branch`
- `git diff --stat`
- `git diff`
- `git diff --staged --stat`
- `git diff --staged`

### 2. Commit if needed

- If there are staged or unstaged changes:
  1. Stage appropriate files with `git add -- <paths>` when needed. Prefer path-specific staging over `git add .`.
  2. Re-check `git status --short` and `git diff --staged`.
  3. Pick a concise commit message based on the staged diff. Follow recent commit style if obvious from `git log --oneline -5`.
  4. Run `git commit -m "<message>"`.
- If there are no local changes, do not create an empty commit. Continue only if there are existing commits to push.

### 3. Push committed work

1. Identify branch and upstream:
   - `git branch --show-current`
   - `git rev-parse --abbrev-ref --symbolic-full-name @{u}`
2. Stop and ask if in detached HEAD or if the target remote/branch is unclear.
3. Review commits to push:
   - With upstream: `git log --oneline @{u}..HEAD`
   - Without upstream: `git log --oneline origin/<current-branch>..HEAD` if possible, otherwise `git log --oneline -5`
4. Push:
   - With upstream: `git push`
   - Without upstream: `git push -u origin <current-branch>` unless the user specified another target.

### 4. Report

Report:

- new commit hash and message, or that no new commit was needed
- remote branch pushed
- whether upstream was set
- any uncommitted changes left behind
</skill>
